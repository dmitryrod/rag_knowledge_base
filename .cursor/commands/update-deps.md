# update-deps

Проверить устаревшие зависимости и обновлять их по одной, прогоняя тесты после каждого шага.

## Цель

Безопасно обновить зависимости: каждая обновляется отдельно, после каждого шага запускаются тесты. При падении тестов можно откатить только одно изменение.

## Шаги

1. **Определить менеджер пакетов**  
   По наличию файлов выбрать: `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` (или `bun.lockb`). Убедиться, что команды ниже соответствуют выбранному менеджеру.

2. **Получить список устаревших зависимостей**  
   - **npm:** `npm outdated` (или `npm outdated --long` для деталей).  
   - **yarn:** `yarn outdated`.  
   - **pnpm:** `pnpm outdated`.  
   Зафиксировать список пакетов (и при желании текущую/желаемую версии).

3. **Обновлять по одному пакету**  
   Для каждого устаревшего пакета по очереди:
   - Обновить только этот пакет до последней разрешённой версии (в рамках semver по `package.json` или до latest — уточнить по контексту).
   - **npm:** `npm update <package>` или `npm install <package>@latest` при необходимости.  
   - **yarn:** `yarn upgrade <package>`.  
   - **pnpm:** `pnpm update <package>`.
   - Сразу после обновления **запустить тесты**. Используй `config.json` → `testing.command`, если задано. Иначе — определи по `package.json`, `pyproject.toml` и т.д. (например `npm test`, `pytest`, `go test ./...`).
   - Если тесты прошли — зафиксировать (коммит по желанию пользователя) и перейти к следующему пакету.
   - Если тесты упали — откатить изменение этого пакета (восстановить версию в lockfile / `package.json` или `npm install` / `yarn install` / `pnpm install`), сообщить пользователю и перейти к следующему пакету.

4. **Итог**  
   Кратко перечислить: какие пакеты обновлены, какие пропущены из-за падения тестов (и по какой причине, если известно).

## Заметки

- Не обновлять несколько пакетов за один шаг — только по одному, затем тесты.
- Если в проекте нет тестов или команда тестов не задана (ни в `config.json`, ни в package.json/pyproject.toml) — предупредить и спросить, продолжать ли только обновления без прогона тестов.
- Опционально: сначала обновлять только patch/minor, major — отдельно или по согласованию с пользователем.
