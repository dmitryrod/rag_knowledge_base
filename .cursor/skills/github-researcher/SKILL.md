# GitHub Researcher

Скилл для **исследования публичных репозиториев GitHub** в рамках оркестрации Cursor: поиск по ключевым словам, выбор кандидатов, чтение README/лицензии/структуры без подмены роли **`researcher`** — инструменты вызывает тот агент, которому делегировали (`Task(researcher=...)` и т.д.).

## Каноническая роль в репозитории

- **GitHub:** поиск репозиториев, issues, релевантных файлов; сравнение звёзд/активности (ориентир, не метрика качества).
- **Не дублировать** `user-context7`: документация библиотек и актуальный API — сначала **context7**; GitHub-скилл — для **поиска проектов**, обзоров, форков, примеров интеграции.
- **Не дублировать** локальную RAG-память ([`PROJECT_RAG_MEMORY.md`](../../docs/PROJECT_RAG_MEMORY.md)): она про **ваши** транскрипты чатов; этот скилл — про **внешние** репозитории.

## Предпочтительный путь: MCP GitHub

Если в сессии доступен сервер **GitHub MCP** (часто имя в списке: `github` или с префиксом `user-`), используй **`call_mcp_tool`** с этим `server` и `toolName` из дескриптора.

1. Перед первым вызовом прочитай JSON-схему нужного инструмента в каталоге MCP descriptors (если доступен): `mcps/<server-name>/tools/<tool>.json`.
2. Типичные сценарии:
   - **Поиск репозиториев** — инструмент уровня `search_repositories` (точное имя зависит от версии сервера; сверяйте с дескриптором).
   - **Содержимое файла** — `get_file_contents` (или эквивалент) для `README.md`, `LICENSE`, ключевых конфигов.
   - **Issues/PR** — по необходимости для оценки зрелости и багов.

**Токен:** `GITHUB_PERSONAL_ACCESS_TOKEN` (classic) с минимально нужными scope для чтения публичных данных; при ошибках авторизации — проверить env в настройках MCP. Секреты не коммитить — см. [`mcp.github.example.json`](../../mcp.github.example.json).

## Запасной путь: Firecrawl MCP

Если GitHub MCP недоступен, используй **`user-firecrawl-mcp`** (см. [`firecrawl-mcp`](../firecrawl-mcp/SKILL.md)):

1. `firecrawl_search` с запросом вида `site:github.com <тема> cursor OR mcp` (или узкая формулировка задачи).
2. Для 2–5 лучших URL — `firecrawl_scrape` (markdown) для README/страницы репо.

Не запускать тяжёлый `crawl` без необходимости.

## Эскалация (узкий → широкий)

1. Нет точного имени репо — поиск по GitHub (MCP или `firecrawl_search`).
2. Нужен обзор одного репо — README + при необходимости `get_file_contents` по путям из выдачи.
3. Нужно сравнить несколько подходов — таблица: репо, лицензия, последняя активность, плюсы/минусы, риски, ссылка на issue.
4. Нужны **официальные** API-доки библиотеки — переключиться на **`user-context7`**, не копать только GitHub.

## Ограничения

- Не выдумывать URL, звёзды, лицензии — только из ответа инструмента или страницы.
- Квоты/ключа Firecrawl — зафиксировать пользователю; не подставлять выдуманные репозитории.
- Крупные ответы не раздувать чат: выдержки, списки ссылок, краткие выводы.
