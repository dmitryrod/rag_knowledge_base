# Planning

Паттерны для декомпозиции задач и создания планов. Применяется агентом `planner`.

## Связь с brainstorming

**Порядок:** сначала скилл **`brainstorming`** (идея → согласованный дизайн, при полном проходе — файл `.cursor/plans/YYYY-MM-DD-<topic>-design.md`), **затем** паттерны ниже. Исключение — **trivial path** из `brainstorming/SKILL.md`: сразу Situation Snapshot и таблица Plan.

Скилл `planning` **не заменяет** brainstorming: он задаёт формат **выхода плана** после того, как границы решения зафиксированы.

## Pattern 1: Situation Snapshot

Перед декомпозицией зафиксируй текущее состояние:

```
Что уже есть:    <файлы, модули, функции в app/ относящиеся к задаче>
Что болит:       <проблема или gap, который нужно закрыть>
Что нельзя трогать: <стабильные интерфейсы, production-данные, контракты API>
Constraints:     <технические ограничения: asyncio, SQLAlchemy, FastAPI>
```

## Pattern 2: Gap-to-Goal Mapping

Для каждой подзадачи явно указывай:

```
ID:       FEAT-001
Gap:      <чего не хватает сейчас>
Goal:     <что должно быть после>
Agent:    worker | refactor | documenter | researcher
Verify:   <как проверить что цель достигнута>
Depends:  <ID других задач, которые должны быть выполнены раньше>
```

## Pattern 3: Micro-Iteration Loop

Для каждой подзадачи в плане применяй цикл:

```
1. Цель     — одно предложение, что нужно сделать
2. Изменения — конкретные файлы в app/ которые затрагиваются
3. Риски    — что может пойти не так (1–3 пункта)
4. Верификация — тест или проверка подтверждающая готовность
5. Следующий шаг — ID следующей задачи или "done"
```

## Pattern 4: Next Prompts

В конце плана добавляй готовые промпты для передачи субагентам:

```markdown
**Для worker (FEAT-001):**
Реализуй <конкретная задача> в файле `app/<path>`.
Контекст: <что уже есть, что нельзя трогать>.
Проверь: <критерий готовности>.

**Для test-runner (после FEAT-001):**
Запусти `python -m pytest app/tests/<test_file>.py`.
Если падают — исправь, сохраняя намерение теста.
```

## Обязательный формат вывода плана

```markdown
## Summary
<одно предложение: что делаем и зачем>

## Constraints
<что нельзя менять>

## Risks
<топ-3 риска с вероятностью>

## Plan
| ID | Задача | Агент | Depends | Verify |
|----|--------|-------|---------|--------|
| FEAT-001 | ... | worker | — | тест |
| FEAT-002 | ... | worker | FEAT-001 | тест |

## Next Prompts
<готовые промпты для первой задачи>
```

## Правила

- Не создавай задачи ради задач — каждая должна закрывать конкретный Gap.
- Максимум 7–9 задач на фичу; больше — признак недостаточной декомпозиции верхнего уровня.
- Зависимости (`Depends`) указывай явно; неявные зависимости приводят к гонкам субагентов.
- Подзадачи без тестов (`Verify = нет`) — только для рефакторинга и документации.
- Подзадачи «собрать внешние источники из веба / сравнить публичные страницы» — агент **`researcher`** (он использует MCP **`user-firecrawl-mcp`** и при необходимости **`user-context7`** по скиллам; в плане указывай роль субагента, а не название MCP).
