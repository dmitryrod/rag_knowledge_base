# Brainstorming → Design → Plan

Адаптация практик [obra/superpowers/brainstorming](https://skills.sh/obra/superpowers/brainstorming) под этот репозиторий. **Цель:** зафиксировать контекст, снять неявные допущения, предложить варианты с компромиссами, получить согласованный дизайн — **до** таблицы задач и вызовов `Task(worker)`.

## Анти-паттерн: «слишком просто для дизайна»

Даже маленькие задачи могут спрятать неверные допущения. Для **нетривиальных** задач (несколько модулей, интеграции, auth/payments, смена контрактов) — **полный чеклист** ниже. Для **тривиальных** (один файл, одна функция, нет развилок) — достаточно **сокращённого прохода**: Situation Snapshot из скилла `planning` + одно предложение цели + сразу переход к формату плана (без отдельного design-файла), с явной пометкой «trivial path».

## Чеклист (порядок обязателен для полного прохода)

1. **Исследовать контекст** — `app/`, `app/docs/ARCHITECTURE.md` при наличии, недавшие изменения; что уже есть, что нельзя ломать.
2. **Визуальный компаньон (опционально)** — если дальше будут **визуальные** вопросы (макеты, сравнение layout, схемы): **отдельным сообщением** предложи пользователю браузерный companion (см. ниже). Не смешивать с вопросами по смыслу.
3. **Уточняющие вопросы** — по одному за раз; по возможности варианты A/B/C; цель: назначение, ограничения, критерий успеха.
4. **2–3 подхода** — с компромиссами и **рекомендацией** одного.
5. **Презентация дизайна** — секции по сложности: границы модулей, поток данных, ошибки, тестирование; после каждой существенной секции — «сходится ли».
6. **Запись спецификации дизайна** — файл в **`.cursor/plans/`** (путь из `config.json` → `documentation.paths.plans`): имя `YYYY-MM-DD-<topic>-design.md`. Содержимое: согласованный дизайн, не дублировать дословно весь чат — только стабильные решения.
7. **Самопроверка спеки** — нет TBD/TODO, нет противоречий, scope один проход реализации; исправить в файле.
8. **Ворота пользователя** — явно: «Спека в `<path>`. Просмотри и скажи, менять ли что-то до плана реализации». Если пользователь в **одном сообщении** дал полный контекст и просит только план — можно записать спеку и сразу спросить подтверждение в том же ответе planner'а, **не** блокируя навсегда сессию (см. «Практика в Cursor»).
9. **Переход к плану** — после согласования дизайна применить скилл **`planning`**: Gap-to-Goal, таблица Plan, Next Prompts. **Не** подменять следующий шаг скиллами реализации (`worker` и т.д.) — это делает родительский workflow через **`Task`**.

## Визуальный companion

Текст из оригинального скилла: предложение **только** в отдельном сообщении, без другого содержимого:

> «Часть вопросов проще показать в браузере (макеты, схемы, сравнение вариантов). Могу собрать визуалы по ходу; это токенозатратно. Подключить? (нужен локальный URL / браузерный инструмент)»

Если пользователь отказал — только текст. **Критерий:** «пользователю легче *увидеть*, чем прочитать?» — да → браузер; нет → терминал/чат.

## Связь с `planning` и `Task`

- **Brainstorming** заканчивается **готовым дизайном** (и при полном проходе — файлом в `.cursor/plans/`) и явным переходом к формату из `planning/SKILL.md`.
- Оригинальный skill завершает вызовом **writing-plans**; в этом проекте его заменяет **скилл `planning`** + обязательный markdown-план.
- Делегирование реализации — только **`Task(subagent_type=...)`** по канону [`.cursor/docs/CREATING_ASSETS.md`](../../docs/CREATING_ASSETS.md#task-delegation).

## Практика в Cursor

- **Planner** вызывается через `Task` из `/workflow-feature` и `/norissk` при выборе feature-workflow: в промпте на planner можно требовать «полный brainstorming → затем план».
- Если **один общий чат** без отдельных сообщений пользователю, planner **всё равно** выполняет шаги 1–5 и 6–7 внутри своего ответа; шаг 8 — короткое приглашение к правкам спеки **или** «если ок — ниже план» в конце того же ответа.

## Запреты

- Не начинать декомпозицию **Gap-to-Goal** до завершения фазы дизайна (кроме trivial path).
- Не писать код и не вызывать `worker` внутри роли planner — только план и рекомендации субагентов.
